# Postmortem — Demo App CrashLoop Incident

**Incident ID:** INC-001
**Date:** 2026-03-09
**Duration:** 12 minutes (22:30 – 22:42 UTC)
**Severity:** SEV-2 (Partial outage)
**Status:** Resolved
**Author:** Platform Engineering

---

## Summary

The `demo-app` deployment entered a `CrashLoopBackOff` state after a bad deployment that set an invalid container command. The `PodCrashLooping` alert fired within 5 minutes. The issue was resolved by rolling back the deployment using `kubectl rollout undo`.

**MTTR (Mean Time to Resolve):** 7 minutes after detection

---

## Root Cause

A misconfigured deployment override set the container command to `["sh", "-c", "exit 1"]` instead of the correct `node src/app.js`. This caused the container to exit immediately on every start, triggering Kubernetes' exponential backoff restart loop.

## Detection

- **Alert:** `PodCrashLooping` (PrometheusRule in `monitoring` namespace)
- **Condition:** `increase(kube_pod_container_status_restarts_total[15m]) > 3`
- **Notification:** Alertmanager → #incidents Slack channel
- **Runbook:** [crashloop.md](../../sre-observability-stack/runbooks/crashloop.md)

The alert fired as expected within the defined 5-minute window.

---

## Resolution

```bash
# Step 1: Identify affected pod
kubectl get pods -n default | grep CrashLoop

# Step 2: Check logs
kubectl logs demo-app-<pod-id> -n default --previous

# Step 3: Identify it's a config issue (exit 1 command)
kubectl describe pod demo-app-<pod-id> -n default

# Step 4: Rollback
kubectl rollout undo deployment/demo-app -n default

# Step 5: Verify
kubectl rollout status deployment/demo-app -n default
```

## What Went Well

- ✅ Alert fired within expected 5 minutes
- ✅ Alert included runbook link — engineer found fix without escalation
- ✅ Rollback executed in < 2 minutes after root cause found
- ✅ No data loss
- ✅ Monitoring (Grafana) clearly showed restart count spike

---

## What Went Wrong

- ❌ No pre-deployment validation to catch invalid container commands
- ❌ CI pipeline didn't test `docker run` before deployment
- ❌ No `helm test` hook to verify deployment health post-rollout

---

## Lessons Learned

1. **Runbook quality matters** — Having the runbook linked in the alert description reduced MTTR by ~5 minutes
2. **Rollback is the fastest mitigation** — Don't attempt to fix forward during an incident; rollback first
3. **Pre-deploy validation prevents incidents** — A simple `kubectl rollout status --timeout=2m` in the pipeline would have caught this
